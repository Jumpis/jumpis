About

whoami

10년 전, 자신의 업무를 완벽히 자동화해 봇에게 맡기고 몇 년간 출근하지 않았다는 한 개발자의 뉴스를 보았습니다. 그 충격은 제게 '나를 온전히 복제한 자동화 봇을 만들겠다'는 강렬한 목표를 심어주었습니다.

그리고 AI로 전자 업무의 상당 부분이 자동화 가능해진 지금, 그 꿈은 현실에 가까워졌습니다. 저는 현재 6년 차 개발자로서 '에이전틱(Agentic) 개발 환경'에 깊이 몰입해 있습니다. 저의 목표는 단순히 단일 개발 작업을 해내는 것을 넘어, 여러 에이전트를 효율적으로 묶고 플로우를 설계해 프로덕트 운영 전체를 자동화하는 것입니다. openclaw, hermes 같은 오픈소스 에이전트를 워크플로 위에 어떻게 배치할지, 무엇을 자동화하고 무엇을 사람의 손에 남길지 결정하는 것이 제 핵심 역할입니다.

세상에 등장하는 새로운 기술에 대해 "이런 게 있구나" 하고 지나치지 않습니다. 직접 다뤄보고, 제 워크플로에 연동해 직접 장단점들을 체감합니다. 저는 그 과정 위에서 가장 나은 자동화의 해답을 찾아가는 개발자입니다.

Orchestration flow

how it runs
Jumpis (human) Codex PM Codex Dev Claude Review Admin (Jumpis Clone) Safety / approval Artifacts
  1. DEV PATH · main Jumpis → Codex PM → Codex Dev → Claude Review. PM이 PRD/계획/acceptance를 작성하면 Dev가 구현·테스트·디버깅하고, Claude가 정합성·보안·엣지 케이스를 본다. 테스트나 리뷰가 떨어지면 fix loop로 Dev에게 되돌아가고, 통과하면 Jumpis에게 최종 요약을 올린다.
  2. OPS PATH · side Jumpis → Admin (Jumpis Clone). Admin은 개발 PM이 아니라 봇 운영/진단 레이어다. 프로파일·게이트웨이·로그를 점검하고, 문제가 보이면 진단·승인 게이트를 거쳐 안전한 변경만 적용한 뒤 Jumpis에게 짧은 summary와 next action을 돌려준다.
